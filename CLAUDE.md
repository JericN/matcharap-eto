# Matcharap Eto — repo guide

**"Matcharap Eto"** — a Philippine pop-up matcha-latte business by **BriarBear** (a vendor/research board for matcha pop-up events, drink costing, powder sourcing, and competitor study). *(Renamed from the earlier working title "Matcha ni Pou" — that name is dead; use Matcharap Eto everywhere.)* This folder holds research artifacts (root) + the live web app (`web/`).

## Layout
- **`web/`** — the live site: **Next.js 14 (app router) + Tailwind v3**. This is what to edit. It is its **own git repo** → GitHub `JericN/matcharap-eto`, **deployed on Vercel** (auto-detects Next.js). GitHub Pages is disabled. `web/README.md` documents setup.
- Root `*.mjs` / `*.json` / `*.md` / `index.html` — **legacy research & scratch artifacts** (early crawlers, the old single-file site). Do **NOT** edit the old root `index.html` or the orphaned `web/lib/data.js` — both are superseded by `web/src/config/`.

## `web/` architecture — layered, schema-driven, single source of truth
- **Content data = one object: `web/src/config/seed.js`** → `seed = { events, eventLinks, powders, powderImages, competitors, milkOptions, drinks, ingredients, pricing }`. This is the single source of truth for content — read directly by `store.js` (no remote store; Edge Config was removed).
- **Shapes & validation: `web/src/config/schemas.js`** (Zod). `SiteDataSchema.parse()` runs once at the data boundary, so everything above trusts the data. **Edit data ⇒ keep it valid against the schema** (and vice-versa). Shared `LinkSchema` ({kind, url}) is used by `eventLinks` and competitor `links`.
- **Layer 1 (boundary) validates; Layer 2 (features/UI) trusts** — no defensive re-checks in components.
- **Data access = `web/src/config/repo.js` (the DAL — import this, not store/state directly):** reads merge seed content with the Redis state overlay and return already-valid values.
  - Content reads: `repo.events()` (merges `eventLinks`), `repo.powders()`, `repo.competitors()`, `repo.milkOptions()`, `repo.powderImages()`, `repo.drinks()` (overlays per-drink ingredient attachments), `repo.ingredients()` (seed add-ons ∪ user-created, **reference** prices), `repo.costs()` (packaging + additional, overrides applied).
  - State reads: `repo.savedEvents/savedPowders/savedDrinks()`, `repo.srp()`, `repo.priceOverrides()`.
  - Writes (each reads-modifies-writes the one `state` record): `toggleEvent/togglePowder/toggleDrink`, `setSrp/resetSrp`, `setPriceOverride/resetPriceOverride` (keys `"matcha:<powder>" | "milk:<label>" | "ing:<name>"`), `attachIngredient/detachIngredient(drink, ingredient)` (**single-item server-side delta** off the fresh list, so concurrent edits survive — clients never send an absolute array), `addIngredient({name,emoji,price,link})`, `setCosts({packaging?,additional?})`.
- `store.js` = `getSiteData` (reads `seed`, Zod-parsed). `state.js` = shared mutable state (Upstash Redis, client in `redis.js`). `actions.js` = `'use server'` wrappers (client RPC) that call repo + `revalidatePath`.
- `@/*` → `web/src/*`. UI: `web/src/features/{events,calculator,powders,drinks,competitors}/` + shared `web/src/components/` (incl. `components/form/` field library). Routes: `web/src/app/`. Static assets: `web/public/`.

## Conventions (non-obvious — read before editing)
- **Config reads use React `cache()`** (`getSiteData`, `getState`) — per-request memo, fresh on the next request. **Never use a module-level `let cache` global** for data: it pins stale data for the process lifetime and breaks dev refresh.
- **Powder ₱/g is DERIVED, not stored.** The card badge AND the calculator's matcha list parse it from each powder's **`price` string** via `web/src/features/powders/pricing.js` (`perGram` numeric / `perGramLabel` display / `packGrams` / `toMatchaOptions`). → **Always keep a `₱NN/g` token and a `NNg` pack-size token in every powder `price`.** Sweetened mixes are excluded from the calculator via a `/sweeten|mix/` filter.
- **`powderImages`** (in seed): keyed by **exact powder `name`**; value = absolute URL (hotlinked) **or** root-relative `/public` path. No entry ⇒ colored-dot fallback. Renders 56px circle, `object-cover`.
- **`eventLinks`** (in seed): overlay keyed by **exact event `name`** (same pattern as powderImages); merged onto events in `repo.events()`. Each = `[{kind,url}]`. Rendered as a **“sources” chip row** on the EventCard (mirrors the competitor card). Add organizer IG + venue `maps` + `web`-search links; **verify handles live before adding** (see IG SOP).
- **`competitors`** (in seed) = the matcha board — *what the research builds.* **3 `tier`s** (`'general' | 'giant' | 'japan'`, default `'general'`), rendered as 3 stacked **sections** (top→bottom): **🍃 Big Leaves** (`giant` — PH corporate / multi-branch chains: The Matcha Tokyo, Tsujiri, Chotto, SUPER MATCHA), **🌱 Little Leaves** (`general` — the smaller PH homegrown field), **🇯🇵 Straight from Japan** (`japan` — authentic Japan-only houses for product ideas, `region:'japan'`, ¥ prices, a `spotlight:'unique'|'iconic'` corner stamp instead of `threat`). **The Big/Little split is by actual SCALE** (branches/reviews/reach), not premium-ness (MODO is single-store ⇒ `general`). **Tier rule = AVAILABILITY:** Big/Little = brands you can buy/visit IN the Philippines (even foreign ones like Tsujiri/The Matcha Tokyo that have PH branches); the Japan tier is *strictly* houses NOT available in PH (so Ippodo & Marukyu-Koyamaen are excluded — they're sold here). Japan top-10 ≈ 6 `unique` + 4 `iconic`. Per brand: `region` (north/central/south), `format`, `area`, `price` (signature ₱) + `band` (budget/mid/premium); **Maps-verified** `rating`/`reviews`/`open`; **IG-verified** `ig` followers (TikTok counts can't be scraped, so none is stored — TikTok survives only as a `links` entry); `sig`, `menu` [{i,p}] (price-range is **derived** in the card), `sourcing`, `hook`, `scale`, `opened`, **`threat`** (strong/moderate/niche), `healthTxt` (the momentum badge text), optional `note`, and **`links`** [{kind: web|ig|fb|tiktok|maps|order, url}]. Card tags by tier: 🍃 Big Leaf (brown circle) / 🌱 Little Leaf (green) / 🇯🇵 Japan (clay-red circle, ¥, ✨unique|🌟iconic). The grid renders the 3 sections with **area + price filters** (no tier filter — tiers ARE the sections). Research pools live in the root: `competitors.json` (PH) + `japan-matcha.json` (the 33-house Japan pool); `seed.js` is canonical. Build/verify it via the scraping SOP below.
  - **Card** (`CompetitorCard.jsx`): top row = unified all-green pills (the threat/spotlight stamp + `healthTxt`) · a small-circle **✓** verified mark · a **heart** (top-right); then a rank circle + `🌱/🍃/🇯🇵 leaf · format` kicker + name; a full-width `hook` line; the price box (japan shows **`≈₱`** via render-side **`JPY_TO_PHP = 0.379`**, the only place ¥ becomes ₱); the `🍵 sourcing` claim; `📍 area / 📋 menu / 👥 IG·scale` (falls back to `🏠 scale` when `ig` is null) + optional `⚠️ note`; and a bottom **links chip row** (the *only* tappable, pill-shaped elements). IG follower counts are **logged-in-IG-verified**; null ⇒ no count shown (e.g. houses with no official IG).
  - **Heart / save**: persists to `state.savedCompetitors` (`repo.savedCompetitors`/`toggleCompetitor` → `actions.toggleCompetitor` → `revalidatePath('/competitors')`); the page is **`force-dynamic`**. There is **NO separate "hearted" section** — instead hearted brands **sort to the front of their own tier, in heart order** (first-hearted first), so the shortlist is easy to scan; client state is optimistic so a card jumps to the front the instant you heart it.
  - **Grid** (`CompetitorsGrid.jsx`): the 3 tier sections, each a **round-robin masonry** — responsive **1/2/3 columns** by viewport, cards dealt column-by-column so they read **left-to-right** (1,2,3 / 4,5,6 …) with **no row gaps** (each column flows at its own height) — plus the **area + price** filter chip rows.
- **Drinks ↔ ingredients data model.** `ingredients` (seed) are **first-class priced add-ons**, each a self-contained object `{ name, emoji, price, link }` (`price` = ₱/cup reference; `link` = a reference URL or `null`; partial-fill on create is fine). There is **no separate `sources` list** — every reference link lives on its ingredient. `drinks` are `{ name, note, desc, milkMl, ingredients:[names], srp }` (`note` = short subtitle, `desc` = long researched blurb); a drink's add-ons reference ingredients **by name**. **Matcha & milk are removable per-drink "bases"** — `repo.drinks()` overlays `hasMatcha`/`hasMilk` from `state.drinkBases` (absent ⇒ present); the calculator excludes a removed base's cost (a no-milk / matcha-only drink is valid, **no error**). `seed.drinkImages` (keyed by exact drink name) holds self-hosted reference photos `{ src, source, credit }`. The **Drinks page** (`features/drinks/`) renders cards as: title + subtitle, a pill row (green-filled removable **matcha/milk base pills**, outlined **add-on pills**, and a **`+ add` pill** opening a dropdown of removed-bases + unattached ingredients), the long `desc`, and a **photo thumbnail grid → full-screen lightbox** (the lightbox MUST `createPortal` to `document.body` — `.paper-card` has `overflow-hidden` + a hover `transform` that would otherwise trap a `fixed` overlay inside the card). Hearts drinks, attaches/detaches ingredients, toggles bases, and creates ingredients; the catalog is `seed.ingredients ∪ state.extraIngredients`.
- **Drink photos = self-hosted, accuracy-validated.** Real market shots scraped per drink from the **DuckDuckGo image API** (`web/scripts/ddg-drink-images.mjs` — far better relevance than Openverse, which lacked flavor-specific photos, or Bing-via-curl, which returns ad/meme spam for niche queries). Each drink uses **specific** queries (never a generic "matcha latte" fallback — that bled identical images across drinks) with a **title filter** (`"matcha"` + a flavor keyword) and **global dedup by URL + md5** so no two drinks ever share a photo. Downloads are thumbnailed to ≤640px under `web/public/drinks/<slug>/`. **Mandatory step:** build numbered **contact sheets** (`web/scripts/contact-sheet.py`, PIL) and visually inspect each — DDG still lets through the odd product-pack/ad/cosmetic shot — then keep only accurate, distinct ones and splice into `seed.drinkImages` (`{src, source, credit}`, shown in the lightbox).
- **Calculator** (`features/calculator/`): four labelled sections (each `<SecHead>`-titled, big spacing between them) — ① **Recipe** (2 selectors on top: matcha · milk; 3 number fields below: dose g/cup · packaging · additional — all **local** session state, not synced), ② **Ingredient prices** (only the add-ons used by hearted drinks — `usedIngredients` — each `reference + override`), ③ **List of drinks** (cups per drink + per-cup COGS/SRP/profit/margin + a breakdown line that sums exactly; the line omits a removed matcha/milk base), ④ **Totals** (COGS with a per-component breakdown — Σ cups×per-cup matcha/milk/add-ons/packaging/additional — that equals total COGS exactly · revenue · profit · plus a boxed **per-ingredient add-ons breakdown** from `tot.addonByName`). Matcha selection is tracked **by powder label** (stable identity), not a positional index. Margin is **ingredient-COGS only** (excludes booth fee/labour/spoilage). Math is pure in `cost.js` (`cogsForDrink` + `tallyTotals`, both honour `hasMatcha`/`hasMilk`). No batch/shopping-list, no sources section.
- **Form controls = reusable components in `web/src/components/form/`** (`TextField`, `NumberField`, `SelectField`, `RangeField`, `Field`). **Use these, not raw `<input>/<select>`.** CSS skins live in `globals.css @layer components`, all `font-mono` so every input matches: `.field-box` (boxed text/number — **no** chevron), `.field-select` (boxed `<select>` — the **only** field with the dropdown chevron, because it really drops down), `.field-underline` (inline dashed-underline number — cups/SRP/price overrides), `.range-input`. NumberField has `variant="box"|"underline"` + optional `prefix`/`suffix` (e.g. `₱`). **Never put `.field-select` on a non-`<select>`** — the chevron then lies.
- **Theming = CSS variables.** The whole palette is `:root` custom props (RGB triplets) in `web/src/app/globals.css`, mapped into `tailwind.config.js` as `rgb(var(--c-*) / <alpha-value>)`. **To reskin, edit the `:root` block** — every utility / `@apply` / opacity-modifier follows. No hardcoded hex in UI (mascot/logo SVG art is the intentional exception). Hand-drawn primitives (`.paper-card`, `.stamp`, `.chip`, `.card-grid`, form controls…) live in `@layer components`.
- **Mobile**: root font-size steps down in `globals.css @layer base` (15px ≤640px, 14px ≤380px) to scale the rem-based UI; grids use `.card-grid`.

## Shared mutable state (centralized — everyone sees the same)
- **One global Redis `state` record** holds everything mutable (NOT per-user — every visitor reads/writes the same thing). Shape (every field `.default()`-ed, so an empty store parses): `{ savedEvents[], savedPowders[], savedDrinks[], savedCompetitors[], srp{drink→₱}, priceOverrides{"matcha:…"|"milk:…"|"ing:…"→₱}, drinkIngredients{drink→[names]}, drinkBases{drink→{matcha?,milk?}}, extraIngredients{name→{emoji,price,link}}, costs{packaging?,additional?} }`. Read/written in `state.js` (`getState`/`writeState`) via the client in `redis.js`. **Why Redis, not Edge Config:** Edge Config's free tier caps writes at **250/month** — far too low for interactive hearts/SRP saves; Upstash free is 256 MB / 500K cmds/mo.
- **No fallbacks, validate once.** `state.js` **throws loudly** if Redis is unconfigured — there is **NO `.data/state.json` local fallback** (it would silently diverge per-machine). `getState`/`writeState` run `StateSchema.parse()` at the boundary; consumers trust the result and never re-check (no defensive `??`/optional-chaining). Bad data (e.g. a malformed ingredient `link`) is **rejected**, not coerced.
- Env (set in `web/.env.local` for dev **and** the Vercel project env for prod): `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN` (the `KV_REST_API_URL`/`KV_REST_API_TOKEN` pair is also accepted). Provision via Vercel → Storage → Marketplace → **Upstash for Redis**, then Connect to the project (auto-injects the vars). Verify the wiring with **`npm run check:redis`** (set→get→del round-trip).
- `/calculator`, `/powders`, `/drinks`, `/events`, `/competitors` are `export const dynamic = 'force-dynamic'` so they read shared state (hearts/SRP/overrides) fresh per request. Only the static home `/` shows `seed.js` edits on rebuild/redeploy (in dev immediately).

## Verify — ALWAYS
- `cd web && npm run build` — static generation runs the Zod parse, so it **catches bad data, not just bad code.** Want "✓ Compiled" + all routes generated (○ static, ƒ dynamic).
- **Dev server gotcha (important):** run `npm run dev`. **Never run `next build` against the live dev `.next`** — it pollutes webpack's persistent cache → `vendor-chunks/@vercel|zod` resolve warnings + flaky Fast Refresh. If you must prod-build for a check, `rm -rf .next` (or use a clean dir) before the next `next dev`. After a clean start those warnings should be **0**; hot-reload then works (test: edit a server component, `curl` the route, see the change, revert).
- `PageNotFoundError: /_document` or weird stale renders = stale `.next` → `rm -rf web/.next` then restart.
- **Restart dev cleanly.** Rapid simultaneous edits (you + the linter) can corrupt the `.next` chunk manifest → `Cannot find module './NN.js'` 500s on one route. Fix order = **stop `next dev` first, *then* `rm -rf .next`, then restart** — deleting `.next` *under* a running server breaks its in-memory manifest (every route 404s). **Background-launched dev servers (`run_in_background` or `nohup`) get reaped after the turn in this environment** → for a server that persists, **have the user run `npm run dev` in their own terminal.** (`setsid` isn't on macOS; foreground `sleep` is blocked — wait for boot with `curl --retry --retry-delay 2 --retry-connrefused`.)
- **Files change under you mid-task** (collaborator + linter). **Re-Read right before every Edit.** The Edit tool requires the **Read tool** to have opened the file (not `cat`).

## SOP — multi-agent review / cleanup workflow (proven)
For a substantial **review, cleanup, or refactor** where accuracy matters, run a `Workflow` (opt-in — only when the user asks for multi-agent / parallel work) with **redundant agents brainstorming across 5 phases**. Keep it **behavior-preserving** and **scoped to the recently-churned code**; let the guardian + QA be the redundancy, and the **human (main loop) own the final build + visual check**.
1. **Review** — parallel **lens** agents read the ACTUAL code, one lens each: *dead-code · comment-noise · organization · react/next · over-engineering · correctness · consistency · data-integrity* → structured findings (`file · issue · fix · why`). Diverse lenses = redundant coverage; agents must verify by grepping, not guess.
2. **Validate** — one validator **per file** re-checks each finding against the real code (apply / reject — reject behavior or visual changes, over-engineering, scope creep) and picks the safest fix; then a single **guardian** prunes the whole set (kills scope creep, dedupes already-applied no-ops, resolves contradictions).
3. **Spec** — one synthesizer turns the approved set into a single **cross-file-consistent, ordered** edit plan grounded in this file. **Atomic units matter:** e.g. removing a `CompetitorSchema` field MUST also strip that key from every `seed.js` entry *and* every component ref **in the same pass** — `npm run build` runs the Zod parse, so a half-done removal fails the build.
4. **Implement** — **one agent per file** (disjoint files ⇒ parallel-safe, no edit conflicts); apply the spec exactly, nothing more.
5. **QA** — ≥2 redundant static reviewers grep for dead/dangling refs, regressions, and behavior/visual drift. **Do NOT run `next build` inside the workflow** (it corrupts the live dev `.next`) — the main loop runs the authoritative `npm run build` + a render check **afterward**, with the dev server stopped. (A real run: 52 findings → guardian-approved 17 → 4 files edited, build green, zero regressions — the only QA "flags" were a feature the agents weren't told was intentional.)

## SOP — competitor scraping & verification (channel reliability)
Build the board with a **discover (parallel) → verify (sequential) → score** flow:
- **Discover breadth** with parallel WebSearch agents — or a `Workflow` (≈8 search angles → per-brand validate → adversarial verify → weighted score: matcha-focus + indie-comparability + locality + studiability + market-signal, scaled by confidence). Indie pass excludes big chains/organizers; top-brand pass is one agent that excludes the indies. Agents **return `null` rather than guess** a handle/price. Then **dedupe fuzzily** (the same brand surfaces under name variants).
- **Verify each finalist IN-PLATFORM via the Chrome MCP.** The browser is **single-instance → verify sequentially in the main loop** (don't fan it out across agents). **HTTP 200 only proves a URL resolves, not that it's the right/active brand** (socials 200 behind login walls).

**Channel reliability — what actually works (2026):**
- **Google Maps = the workhorse, NO login.** `navigate` to `https://www.google.com/maps/search/?api=1&query=<brand+area>`, then `get_page_text` → ⭐rating, review count, **open / "Permanently closed"**, address (confirms Metro Manila), menu highlights, and a **"people also search for"** list that surfaces *more* competitors. The source of truth for rating / open-status / locality (it caught a brand whose flagship had quietly closed, and review counts the web research had wrong by 25×).
- **Facebook = viewable LOGGED-OUT.** `get_page_text` on `facebook.com/<page>` → follower count, **bio (sourcing / locations / hours)**, Page category, **last-post recency** (activity/momentum), and the **cross-linked IG handle** (confirms it's the right account, not a namesake).
- **Instagram = LOGIN-GATED logged-out** (redirects to `/accounts/login`). Once the **user** is logged into IG in the Chrome session, `get_page_text` on `instagram.com/<h>/` returns `X posts · Y followers · Z following` + bio (sourcing/locations/opened/hours) — or use the `web_profile_info` fetch (next SOP). **Never enter credentials yourself — ask the user to log in.**
- **TikTok = hard anti-bot wall** ("Please wait…") that blocks automated viewing even when other tabs are logged in. Keep the link but **don't display TikTok follower counts**, and **don't attempt to defeat bot-detection / CAPTCHAs.**
- **Link health:** `curl -I` (browser UA) catches dead `web`/`order` links; social URLs 200 even when wrong, so confirm those in-platform.
- **Browser quirks:** `list_connected_browsers` → have the **user pick** (AskUserQuestion) → `select_browser`; tabs get reaped during user interaction (re-create via `tabs_context_mcp({createIfEmpty:true})`); zoom can stick huge (`cmd+0` resets); `find` refs go **stale after a re-render** (re-find right before clicking); a giant-mascot / unstyled flash = the page mid-recompile, not a bug — re-check.

## SOP — verify socials / event dates ON the platform (not Google)
With FB/IG logged into the Chrome MCP session, confirm organizer handles + upcoming dates from the source:
- From an `instagram.com` tab, same-origin fetch the web profile API:
  `fetch('https://www.instagram.com/api/v1/users/web_profile_info/?username=<h>',{headers:{'x-ig-app-id':'936619743392459'},credentials:'include'})`
  → returns `biography`, `edge_followed_by.count`, `external_url`. **Bios reliably carry the next event's dates + “merchant form” links** (the post grid often returns empty).
- A `<meta property="og:title">` on `instagram.com/<h>/` confirms a handle exists.
- **Do small batches (≤2–3 handles per call)** — big loops hit the CDP ~45s renderer timeout.

## SOP — validate that images RENDER (not just HTTP 200)
- Load each image the way the app does — headless Chromium, `<img referrerpolicy="no-referrer">`, check `naturalWidth > 0`. **Playwright lives in the root `node_modules`** → run scripts with `NODE_PATH=/Users/jeric/Documents/GitHub/Matcha/node_modules node <script>` (or import via absolute path).
- Prefer **Shopify/store CDN** URLs (`cdn.shopify.com`, store `/cdn/shop/...`) — they hotlink reliably with no-referrer. Decode `&amp;`→`&` from HTML-scraped URLs.
- No durable hotlink (Shopee/Lazada/IG-only, signed/expiring)? **Download + self-host** under `web/public/` and reference a root-relative path (the `powderImages` schema accepts URL **or** path).
- `file://` render checks return 0×0 (cross-origin sandbox) — validate over **HTTP** (`cd web/public && python3 -m http.server`, then load the http URL).

## SOP — matcha pricing research
Verify prices from each brand's **official store**: Shopify `<product>.js` / `<product>.json` gives exact variant price/grams/availability. Flag stock (many PH boutique brands are perpetually sold out). FX ≈ ₱58–61/USD (brands price in USD/JPY; PHP is a conversion — note it).
